<h1><a id="user-content-using-expressroute-for-routing-microsoft-365-traffic" class="anchor" aria-hidden="true" href="#using-expressroute-for-routing-microsoft-365-traffic"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Using ExpressRoute for Routing Microsoft 365 Traffic</h1>
<p>An ExpressRoute circuit provides private connectivity to Microsoft backbone network.</p>
<ul>
<li>It offers <em>Private peering</em> to connect to private endpoints of your IaaS deployment in Azure regions</li>
<li>Also, it offers <em>Microsoft peering</em> to connect to public endpoints of IaaS, PaaS, and SaaS services in Microsoft network.</li>
</ul>
<p>For more information about ExpressRoute, see the <a href="https://docs.microsoft.com/azure/expressroute/expressroute-introduction" rel="nofollow">Introduction to ExpressRoute</a> article.</p>
<p>Often there's a confusion whether ExpressRoute can be used or not for routing Microsoft 365 SaaS traffic.</p>
<ul>
<li>One side argument is: ExpressRoute does offer Microsoft peering, using which you can reach most of the public endpoints in Microsoft network.
In fact, using a <em>Route Filter</em> you can select Microsoft 365 service prefixes that need to be advertised via Microsoft peering to your on-premises network.
These routes advertisement enables routing Microsoft 365 service traffic over the ExpressRoute circuit.</li>
<li>The counter argument is: Microsoft 365 is a distributed service. It is designed to enable customers all over the world to connect to the service using the Internet.
So, it's recommended not to use ExpressRoute for Microsoft 365.</li>
</ul>
<p>The goals of this article are:</p>
<ul>
<li>to provide technical reasoning for the arguments, and</li>
<li>objectively discuss when to use ExpressRoute for routing Microsoft 365 traffic and when not to use it.</li>
</ul>
<h2><a id="user-content-network-requirements-of-microsoft-365-traffic" class="anchor" aria-hidden="true" href="#network-requirements-of-microsoft-365-traffic"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Network Requirements of Microsoft 365 Traffic</h2>
<p>Microsoft 365 service often includes real-time traffic such as voice &amp; video calls, online meetings, and real-time collaboration. This real-time traffic has stringent network performance requirements in terms of latency and jitter. Within certain limits of network latency, jitter can be effectively handled using buffer at the client device. Network latency is a function of physical distance traffic need to travel, link bandwidth, and network processing latency.</p>
<h2><a id="user-content-network-optimization-features-of-microsoft-365" class="anchor" aria-hidden="true" href="#network-optimization-features-of-microsoft-365"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Network Optimization Features of Microsoft 365</h2>
<p>Microsoft strives to optimize network performance of all the cloud applications both in terms of architecture and features. To begin with, Microsoft owns one of the largest global networks, which is optimized to achieve the core objective of offering best network performance. Microsoft network is software defined, and it's a "Cold Potato" network. "Cold Potato" network in the sense, it attracts and egress traffic as close as possible to client-device/customer-network. Besides, Microsoft network is highly redundant and highly available. For more information about architecture optimization, see <a href="https://azure.microsoft.com/blog/how-microsoft-builds-its-fast-and-reliable-global-network/" rel="nofollow">How Microsoft builds its fast and reliable global network</a>.</p>
<p>To address the stringent network latency requirements, Microsoft 365 shortens route length by:</p>
<ul>
<li>dynamically routing the end-user connection to the nearest Microsoft 365 entry point, and</li>
<li>from the entry point efficiently routing them within the Microsoft global network to the nearest (and authorized) Microsoft 365 data center.</li>
</ul>
<p>The Microsoft 365 entry points are serviced by Azure Front Door (AFD). AFD is a widely distributed service present at Microsoft global edge network and it helps to create fast, secure, and highly scalable SaaS applications. To further understand how AFD accelerates web application performance, see <a href="https://docs.microsoft.com/azure/frontdoor/front-door-overview" rel="nofollow">What is Azure Front Door?</a>. While choosing the nearest Microsoft 365 data center, Microsoft does take into consideration data sovereignty regulations within the geo-political region.</p>
<h2><a id="user-content-what-is-geo-pinning-connections" class="anchor" aria-hidden="true" href="#what-is-geo-pinning-connections"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>What is Geo-Pinning Connections?</h2>
<p>Between a client-server when you force the traffic to flow through certain network device(s) located in a geographical location, then it's referred to as geo-pinning the network connections. Traditional network architecture, with the underlying design principle that the clients-servers are statically located, commonly geo-pins the connections.
For example, when you force your enterprise Internet connections traverse through your corporate network, and egress from a central location (typically via a set of proxy-servers or firewalls), you're geo-pinning the Internet connections.</p>
<p>Similarly, in SaaS application architecture if you force route the traffic through an intermediate datacenter (for example, cloud security) in a region or via one or more intermediate network devices (for example, ExpressRoute) in a specific location then you're geo-pinning the SaaS connections.</p>
<h2><a id="user-content-when-not-to-use-expressroute-for-microsoft-365" class="anchor" aria-hidden="true" href="#when-not-to-use-expressroute-for-microsoft-365"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>When not to use ExpressRoute for Microsoft 365?</h2>
<p>Because of its ability to dynamically shorten the route length and dynamically choose the closest server datacenter depending on the location of the clients, Microsoft 365 is said to be designed for the Internet.
Besides, certain Microsoft 365 traffic is routed only through the Internet.
When you have your SaaS clients widely distributed across a region or globally, and if you geo-pin the connections to a particular location then you are forcing the clients further away from the geo-pined location to experience higher network latency.
Higher network latency results in suboptimal network performance and poor application performance.</p>
<p>Therefore, in scenarios where you have widely distributed SaaS clients or clients that are highly mobile, you don't want to geo-pin connections by any means including forcing the traffic through an ExpressRoute circuit in a specific peering location.</p>
<h2><a id="user-content-when-to-use-expressroute-for-microsoft-365" class="anchor" aria-hidden="true" href="#when-to-use-expressroute-for-microsoft-365"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>When to use ExpressRoute for Microsoft 365?</h2>
<p>The following are some of the reasons why you may want to use ExpressRoute for routing Microsoft 365 traffic:</p>
<ul>
<li>Your SaaS clients are concentrated in a geo-location and the most optimal way to connect to Microsoft global network is via ExpressRoute circuits</li>
<li>Your SaaS clients are concentrated in multiple global locations and each location has its own ExpressRoute circuits that provide optimal connectivity to Microsoft global network</li>
<li>You're required by law to route cloud-bound traffic via private connections</li>
<li>You're required to route all the SaaS traffic to a geo-pinned centralized location (be it a private or a public datacenter) and the optimal way to connect the centralized location to the Microsoft global network is via ExpressRoute</li>
<li>For some of your static SaaS clients only ExpressRoute provides optimal connectivity, while for the other clients you use Internet</li>
</ul>
<p>While you use ExpressRoute, you can apply the route filter associated with Microsoft peering of ExpressRoute to route only a subset of Microsoft 365 services and/or Azure PaaS services over the ExpressRoute circuit. For more information, see <a href="https://docs.microsoft.com/azure/expressroute/how-to-routefilter-portal" rel="nofollow">Tutorial: Configure route filters for Microsoft peering</a>.</p>
<h2><a id="user-content-next-steps" class="anchor" aria-hidden="true" href="#next-steps"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Next Steps</h2>
