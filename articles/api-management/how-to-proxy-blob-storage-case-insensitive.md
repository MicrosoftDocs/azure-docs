# Use API Management as a case-insensitive proxy for Azure Blob Storage

Azure Blob Storage treats blob paths as case-sensitive. When serving static content with URLs that may vary in casing (for example, GUIDs generated by upstream systems), requests for `/PATH/INDEX.HTML` and `/path/index.html` resolve to different blobs. This article shows how to use Azure API Management (APIM) as a reverse proxy that normalizes URL casing before forwarding requests to Blob Storage.

## Scenario

You have static content stored in Azure Blob Storage (in the `$web` container for static website hosting) and need to:

- Serve content regardless of URL casing in the request
- Append `index.html` to requests ending with a trailing slash
- Authenticate to storage using managed identity (no shared keys)
- Maintain VNet-level network isolation

## Prerequisites

- An Azure API Management instance (StandardV2 or higher for VNet integration)
- An Azure Storage account with static website hosting enabled (`$web` container)
- Content uploaded to the `$web` container

## Configure the API Management policy

The following policy performs three operations on every inbound request:

1. Converts the URL path to lowercase using `ToLower()`
2. Appends `index.html` to paths ending with `/`
3. Acquires a managed identity token scoped to Azure Storage
4. Rewrites the URI to target the `$web` container in Blob Storage

### Complete policy

```xml
<policies>
  <inbound>
    <base />
    <set-variable name="loweredPath" value="@(context.Request.Url.Path.ToLower())" />
    <set-variable name="finalPath" value="@{
      var p = (string)context.Variables["loweredPath"];
      return p.EndsWith("/") ? p + "index.html" : p;
    }" />
    <authentication-managed-identity resource="https://storage.azure.com/" />
    <rewrite-uri template="@("$web" + (string)context.Variables["finalPath"])" />
  </inbound>
  <backend>
    <base />
  </backend>
  <outbound>
    <base />
  </outbound>
  <on-error>
    <base />
  </on-error>
</policies>
```

## Configure the API definition

When creating the API in APIM, set the **Web service URL** (also called `serviceUrl`) to point to the Blob Storage endpoint *without* the `$web` container path:

```
https://<storage-account-name>.blob.core.windows.net
```

> [!IMPORTANT]
> Do **not** include `/$web` in the Web service URL. The `rewrite-uri` policy prepends `$web` to the path. If you include `$web` in both the service URL and the rewrite expression, the resulting backend request will contain a doubled path (`$web/$web/...`) and return 404 BlobNotFound.

This interaction between `serviceUrl` and `rewrite-uri` is the most common source of errors with this pattern. The service URL forms the base of the backend request, and `rewrite-uri` replaces the path portion. They concatenate — they do not override each other.

### Create a wildcard operation

Create a single API operation that catches all paths:

| Setting | Value |
|---------|-------|
| Display name | Catch-all |
| URL | GET `/*` |

This ensures any inbound path is processed by the policy regardless of depth or structure.

## Configure managed identity

1. Enable system-assigned managed identity on the APIM instance
2. Assign the **Storage Blob Data Reader** role to the APIM managed identity on the storage account
3. Allow time for RBAC propagation (up to 5 minutes in practice, though Microsoft documents up to 10)

The `authentication-managed-identity` policy element in the inbound section acquires a token scoped to `https://storage.azure.com/` and attaches it to the backend request automatically.

## Network configuration

If your storage account uses firewall rules (recommended), you must allow the APIM subnet:

- APIM StandardV2 must be VNet-integrated into a subnet
- Add that subnet as an allowed virtual network rule on the storage account
- Storage should have `defaultAction: Deny` with explicit VNet rules

## Troubleshooting

### 404 BlobNotFound

Check the `AzureDiagnostics` table in Log Analytics (not `ApiManagementGatewayLogs`) with this query:

```kusto
AzureDiagnostics
| where ResourceProvider == "MICROSOFT.APIMANAGEMENT"
| where TimeGenerated > ago(30m)
| project TimeGenerated, requestUri_s, backendUrl_s, responseCode_d, backendResponseCode_d
| order by TimeGenerated desc
| take 10
```

The `backendUrl_s` column shows the exact URL that APIM sent to storage. If you see a doubled path (for example, `$web/$web/path`), your service URL includes `/$web` and must be corrected.

### 403 AuthorizationPermissionMismatch

- Verify the managed identity has **Storage Blob Data Reader** (not just Reader) on the storage account
- Wait 5 minutes after role assignment for RBAC propagation
- Confirm the `authentication-managed-identity` element is present in the inbound policy and uses resource `https://storage.azure.com/`

### 401 on backend but 200 expected

If the APIM gateway returns a 401 from storage, the managed identity token may not be attaching. Confirm that:
- System-assigned managed identity is enabled on the APIM resource
- The policy includes `<authentication-managed-identity resource="https://storage.azure.com/" />` before the `<rewrite-uri>` element
- The request is not being short-circuited by a `<return-response>` or `<send-request>` before reaching the backend

## Validation

Test the following URL patterns through your APIM gateway to confirm correct behavior:

| Request | Expected behavior |
|---------|-------------------|
| `GET /path/file.html` | Returns blob at `$web/path/file.html` |
| `GET /PATH/FILE.HTML` | Normalized to lowercase, returns same blob |
| `GET /path/` | Appends `index.html`, returns `$web/path/index.html` |
| `GET /PATH/` | Normalized and appended, returns `$web/path/index.html` |
| `GET /nonexistent/file.html` | Returns 404 |
